Mirage

A study in visually rich and lightning-fast high resolution images for the web.

Problem

Performance is a fundamental component of the user experience, people are impatient and won’t wait for a slow website.

Whilst the idea of an average page is a myth, the reality is that websites are growing larger by the month. High performance should be a requirement of any web project, not an afterthought.

Poor performance has been tied to a decrease in revenue, traffic, conversions, and overall user satisfaction. Case study after case study shows that improving performance, even marginally, will impact the bottom line. The situation is no different on mobile where 71% of people say they expect sites to load as quickly or faster on their phone when compared to the desktop.

A study by Akamai show that 40% of people will abandon a website that takes more than 3 seconds to load.

Meanwhile, images account for 60% (1.8MB) of the assets required to display an average page and are often not resized or optimised.

Challenge

How do we deliver a visually rich website that loads fast enough for all its visitor?

With the growing number of devices, browsers, resolutions and networks we need to make sure we can deliver lightning-fast visually rich websites to everyone, from people with Internet Explorer 10 to people using the latest high-resolution smartphone on 3G.

What’s fast enough?

Use the 20% rule. People perceive tasks as faster or slower than other tasks when the difference in time is at least 20% slower or faster.

This means, if a competitors website loads in 5 seconds, our website needs to load in 4 seconds.

In lieu of that, deliver First Meaningful Paint in less than 1 second, aim for a SpeedIndex value under 1250, make sure the Time to Interactive (TTI) is less than 5 seconds on slow 3G and for repeat visits get the TTI to less than 2 seconds.

Use Webpagetest or Google Lighthouse to test performance.

But with high-resolution devices becoming more common, fast performance is no excuse for visually poor websites, we need to deliver both.

Solution

Use resized to image-matching aspect ratio and optimised low-quality image placeholders (LQIP). Lazy load the device-contextual resolution when the image reaches the viewport. Resize source images to twice the rendered dimensions and optimise the shit out of them.

We’re using the picture element because clients may need explicit control over which image is shown at set viewport sizes. Unlike the srcset and sizes attributes in the img element which instead gives the browser a list of image sources that are identical and let’s the browser choose the best one.

Optimisation

Images are resized using Bilinear resampling and optimised with JPEGOptim and JPEGTran using Huffman code tables with quality set at 40%.

Base64 encoding LQIPs

Base64 encoding an image will increase its file size by 30%.

But when accounting for network latency, which is 200ms on 3G networks for every HTTP single request and the overhead (800-900 bytes) of that request, the benefits still outweigh the drawbacks.

Gotchas

There’s always a catch or twelve, and here they are.

We have to include a img fallback. E.g. IE 11 doesn't recognise the picture element.

We have to include a noscript fallback for circumstances where JavaScript isn’t available.

To support IE9, we need to wrap a video element wrapper around the source elements in the picture element. You can do this using conditional comments as illustrated in all scenarios.

Edge versions 13, 14 and 15 intermittently display distorted images as soon as they encounter a srcset attribute, seemingly dependent on caching and network timing. See bug.

See a list of other unresolved issues and features within the picture specification.

Iconography with type attributes

The type attribute to the rescue.

The browser will pick the first source where the declared image type is one that it supports. If it doesn’t recognize any of the source types, it will use the element’s src or srcset declarations.
